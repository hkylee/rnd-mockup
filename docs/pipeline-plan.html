JSON → TSX → Mockup 자동화 플랜

2026-05-19 make-cmp · make-tsx · make-mockup Figma Code Connect 연동
01 · 경위
왜 이 파이프라인을 만들게 됐나

기존에는 Figma 마스터 컴포넌트 → JSON 명세까지만 자동화되어 있었다 (/make-cmp). 이 JSON은 컴포넌트의 레이아웃·색상·variants 정보를 담고 있지만, 실제로 화면을 그리거나 개발자에게 코드를 넘겨주려면 다음 두 가지가 없었다.

첫째, 화면 미리보기(mockup) — JSON만으로는 컴포넌트가 실제로 어떻게 보이는지 시각 검증이 안 됐다. HTML로 직접 변환하면 Figma의 auto-layout, token alias 등이 어색하게 렌더링되는 문제가 있었다.

둘째, Figma 핸드오프 — 개발자가 Figma에서 컴포넌트를 클릭했을 때 실제 사용할 수 있는 TSX 코드가 보여야 한다. 이를 위해 Figma Code Connect (generate-connect.js) 가 이미 있었지만, 연결할 TSX 파일 자체가 없어서 import path가 <TODO>로 막혀 있었다.

이 두 문제를 해결하기 위해 JSON → TSX 자동 생성(/make-tsx)을 파이프라인 중간 단계로 추가하기로 결정했다. TSX가 생기면 mockup도 정확하게 렌더링되고, Code Connect의 TODO도 자동으로 채워진다.

Before — 막혀있던 것들
  • JSON만으로 화면 렌더링 시 레이아웃·토큰 어색함
  • generate-connect.js의 import path가 <TODO>
  • 개발자 핸드오프 시 실제 TSX 코드 없음
  • make-mockup이 JSON에 직접 의존 → 렌더 품질 낮음
After — make-tsx 추가로 해결
  • JSON → TSX로 정확한 React 컴포넌트 생성
  • Code Connect가 TSX import path 자동 주입
  • Figma 검사 패널에서 실제 코드 바로 확인
  • make-mockup이 TSX를 재사용 → 정확한 렌더링

03 · 단계별 상세
각 스킬이 하는 일
✓
make-cmp — Figma API → JSON 명세 추출
Figma의 마스터 컴포넌트를 API로 조회해 레이아웃·색상·variants·자식 구조를 docs/input/components/json/{name}.json으로 저장. figmaNodeId와 figmaFileKey도 함께 기록해 이후 단계에서 활용.
★
make-tsx — JSON → React TSX 컴포넌트 생성 (이번 완료)
JSON의 base.layout → CSS flex, base.visual → background/border, variants.axes → TypeScript union props로 변환. 출력: docs/input/components/tsx/{name}.tsx. data-figma attribute로 figmaNodeId 임베드 → Code Connect 연결 포인트.
✓
generate-connect.js — TSX + figmaNodeId → Code Connect 파일 생성
이미 구현된 스크립트. make-tsx 완료 후 node scripter/generate-connect.js --all 실행하면 scripter/connect/{name}.figma.tsx 생성. 이전에는 import path가 <TODO>였지만 TSX 파일이 생기면 자동으로 채워짐. 이후 figma connect publish로 Figma에 올리면 핸드오프 완성.
▶
make-mockup — TSX 컴포넌트 조합 → 화면 HTML 생성 (다음 플랜)
specs-input/의 기능내역서 또는 page spec을 읽어 TSX 컴포넌트를 조합, 실제 화면처럼 보이는 HTML 미리보기를 생성. JSON 직접 변환 대신 TSX 경유이므로 레이아웃·토큰이 정확하게 반영됨. 아래 섹션에서 상세 플랜.

07 · Figma 핸드오프 시나리오
마스터 컴포넌트가 이미 있을 때 — 어떤 방식이 맞나

이 프로젝트는 Figma에 마스터 컴포넌트·토큰 Variables·Props가 이미 세팅된 상태다. 이 상황에서 코드 → Figma 방향으로 핸드오프할 때 방식에 따라 결과가 완전히 달라진다.

시나리오 A · 비추천
HTML to Figma
/ Anima
렌더된 HTML → Figma 레이어 변환
작동 방식

브라우저에서 렌더된 HTML/CSS를 캡처해 Figma 레이어로 변환. 시각적으로 비슷한 Frame이 생성됨.

문제점
  • 기존 마스터 컴포넌트의 인스턴스가 아닌 flat 레이어로 들어옴
  • 토큰 Variable 연결 끊김 (raw hex로 변환됨)
  • Props / variants 정보 소실
  • 마스터 업데이트 시 인스턴스 자동 반영 불가
  • 디자이너가 수정할 수 없는 픽셀 덩어리
시나리오 B · 컴포넌트 핸드오프
Figma Code Connect
TSX ↔ 마스터 컴포넌트 연결
작동 방식

.figma.tsx 파일로 TSX 컴포넌트와 Figma 마스터를 1:1 연결. 개발자가 Figma에서 컴포넌트 클릭 시 실제 TSX 코드·props 확인.

장점
  • 마스터·토큰·Props 건드리지 않음
  • 개발자 핸드오프 품질 최고
  • generate-connect.js 이미 구현됨
  • 컴포넌트 단위 연결 — atom/mol/ogn 모두 가능
시나리오 C · 화면 핸드오프
page spec JSON
→ bundle.js
마스터 인스턴스로 화면 Frame 자동 생성
작동 방식

화면 구성을 JSON으로 정의 → bundle.js → Scripter → Figma에 마스터 컴포넌트의 인스턴스로 Frame 구성.

장점
  • 토큰 Variable 연결 살아있음
  • 마스터 업데이트 → 인스턴스 자동 반영
  • 디자이너가 편집 가능한 실제 Figma 화면
  • mockup/component-specs/page/ 이미 이 방식 사용 중

08 · 결론
최적 워크플로우 — B + C 조합

마스터 컴포넌트·토큰이 이미 Figma에 있으므로, 핸드오프는 인스턴스 기반이어야 한다. 시나리오 A(HTML to Figma)는 제외. 컴포넌트 단위는 B, 화면 단위는 C로 나눠 처리하는 것이 최적.

최적 워크플로우
Figma 마스터 컴포넌트 (토큰·Props·variants 세팅됨)
  │
  ↓  /make-cmp  ─────────────────────── 토큰 참조 보존 ({semantic.color.brand.primary})
  │
component spec JSON  (docs/input/components/json/)
  │
  ↓  /make-tsx  ─────────────────────── CSS Variable + TypeScript props 매핑
  │
TSX 컴포넌트  (docs/input/components/tsx/)
  │
  ├──── generate-connect.js  ──────────── 시나리오 B: 컴포넌트 단위 핸드오프
  │       ↓
  │     .figma.tsx  →  figma connect publish
  │       ↓
  │     Figma 검사 패널: 실제 TSX 코드 + props 표시
  │
  ├──── /make-mockup ──────────────────── HTML preview (시각 검증)
  │       ↓
  │     docs/output/mockup/{SCREEN}.html
  │
  ├──── /make-mockup ──────────────────── 시나리오 C: 화면 단위 핸드오프
  │       ↓ (storyboard → page spec JSON)
  │     docs/output/pages/{MODULE}/{screen}.json
  │       ↓  bundle.js → Scripter
  │     Figma Frame (마스터 인스턴스 조합, 토큰·Props 살아있음)
  │
  └──── Web 구현 ──────────────────────── 동일 TSX → Next.js 서비스
          ↓
        web/src/components/library/
B · Code Connect — 컴포넌트 핸드오프
개발자가 Figma에서 badge, button 등 개별 컴포넌트를 클릭했을 때
실제 TSX 코드와 props 사용법을 바로 확인할 수 있게 연결.
generate-connect.js가 이미 구현됨 — make-tsx 완료 후 바로 실행 가능.
C · bundle.js — 화면 핸드오프
storyboard를 인풋으로 화면 Frame을 Figma에 자동 생성.
마스터 인스턴스로 구성되므로 토큰 Variable·variants 살아있음.
디자이너가 편집 가능 + 마스터 변경 시 인스턴스 자동 반영.
핵심 원칙
Figma 마스터는 항상 원본. 코드는 그 마스터를 추출해서 쓰거나 인스턴스로 조합할 뿐, 마스터를 덮어쓰거나 픽셀로 대체하지 않는다.
→ HTML to Figma 방식은 이 원칙에 위배되므로 이 프로젝트에 적합하지 않다.
