# 화면설계서(SB) 작성 가이드

> 기획자를 위한 SDUI 기반 화면설계서 작성 규칙
> Screen-SB, Section-SB, Organism-SB 템플릿 작성 방법

---

## 1. ID 네이밍 컨벤션

### 1.1 화면 ID (Screen)

```
NOVA-{MODULE_ID}-{TYPE_CODE}-{NNN}-{N}
```

| 필드 | 설명 | 규칙 |
|------|------|------|
| `NOVA` | 시스템 prefix | 고정값 |
| `MODULE_ID` | 기능 모듈 코드 | 대문자, 2~6자 ([모듈 목록](module-id-list.md) 참조) |
| `TYPE_CODE` | 구현 유형 코드 | 대문자 2자 (아래 1.2절 참조) |
| `NNN` | 3자리 순번 | 001~999, 모듈 내 순서 기준 |
| `N` | 서브 버전 | 0=기본, 1+=변형/A·B 테스트 분기 |

**예시:** `NOVA-MBR-PG-001-0` (MBR 모듈 첫 번째 Page 화면, 기본 버전)

---

### 1.2 구현 유형 코드 (TYPE_CODE)

#### 페이지 유형

| 코드 | 유형명 | 설명 | 예시 |
|------|--------|------|------|
| `PG` | Page | 전체 화면 페이지 (하단 네비게이션 포함) | 회원가입, 상품 목록 |
| `WV` | WebViewPage | 인앱 웹뷰 전체 화면 | 약관 상세, 이벤트 랜딩 |
| `FM` | FullModal | 전체 화면 모달 (닫기 버튼, 하단 네비 없음) | 개인정보 입력, 인증 단계 |
| `OB` | OnboardingPage | 온보딩/인트로 슬라이드 페이지 | 최초 실행 가이드 |

#### 팝업 유형

| 코드 | 유형명 | 설명 | 예시 |
|------|--------|------|------|
| `CP` | ConfirmPopup | 확인/취소 2버튼 확인 팝업 | "탈퇴 철회하시겠습니까?" |
| `AP` | AlertPopup | 확인 1버튼 알림 팝업 | "오류가 발생했습니다" |
| `IP` | InputPopup | 입력 필드 포함 팝업 | 비밀번호 재입력, 코드 입력 |
| `MP` | MultiPopup | 3버튼 이상 또는 리스트 선택 팝업 | 정렬 선택, 다중 옵션 |

#### 시트/패널 유형

| 코드 | 유형명 | 설명 | 예시 |
|------|--------|------|------|
| `BS` | BottomSheet | 스크롤 가능한 하단 슬라이드 시트 | 필터, 상세 정보 시트 |
| `BF` | BottomSheetFixed | 고정 높이 하단 시트 (스크롤 없음) | 간단 선택, 단계 진행 버튼 시트 |
| `AS` | ActionSheet | 액션 선택 목록 시트 | 공유, 더보기 액션 목록 |
| `DW` | Drawer | 사이드 드로어 패널 | 필터 사이드바, 메뉴 드로어 |

#### 오버레이/기타 유형

| 코드 | 유형명 | 설명 | 예시 |
|------|--------|------|------|
| `TS` | Toast | 자동 소멸 토스트/스낵바 | "저장되었습니다" |
| `TT` | Tooltip | 앵커 기반 툴팁 설명 레이어 | "이 할인은 첫 가입자만 적용돼요" |
| `LD` | LoadingOverlay | 처리 중 전체 로딩 오버레이 | 결제 처리 중, 회원 처리 중 |
| `OV` | GuideOverlay | 기능 설명 코치마크/가이드 오버레이 | 신기능 안내, 첫 방문 가이드 |

### 1.3 섹션 ID (Section)

```
section-{module-id}-{component-name}-{업무코드}
```

**예시:**
- `section-PRDD-list-bundle` → 상품상세 묶음 추천 리스트 섹션
- `section-MBR-auth-form` → 회원 인증 폼 섹션

---

### 1.4 오가니즘 ID (Organism) — 모듈 종속

```
ogn-{module-id}-{component-name}-{variant}
```

**예시:**
- `ogn-PRDD-list-group` → 상품상세 그룹 추천 리스트
- `ogn-MBR-auth-method-selector` → 회원 인증 방법 선택기

---

### 1.5 몰리큘 ID (Molecule) — 범용 재사용

```
mlc-{component-name}-{variant}
```

**예시:**
- `mlc-prd-rcm-banner` → 상품 추천 배너 (모든 모듈에서 사용 가능)
- `mlc-prd-list-period` → 기간별 상품 리스트

---

### 1.6 아톰 ID (Atom) — 범용 재사용

```
atom-{component-name}-{variant}
```

**예시:**
- `atom-typo-display` → 디스플레이 텍스트
- `atom-btn-primary` → 주요 버튼

## 2. 파일 구조 규칙

```
storyboard/
├── {MODULE_ID}/
│   ├── screen/      → NOVA-{MODULE_ID}-{TYPE_CODE}-{NNN}-{N}.html
│   ├── section/     → section-{MODULE_ID}-{경로-dashes}.html
│   └── organism/    → ogn-{MODULE_ID}-{organism-name}.html
```

**예시 (MBR 모듈):**
- `storyboard/MBR/screen/NOVA-MBR-PG-001-0.html`
- `storyboard/MBR/section/section-MBR-auth.html`
- `storyboard/MBR/organism/ogn-MBR-auth-method-selector.html`

## 3. SDUI 필수 필드 작성 가이드

### 3.1 노출 조건

**SDUI 매핑:** `display.when` 필드

#### 작성 방법

| 조건 유형 | 작성 방법 | 예시 |
|----------|----------|------|
| 데이터 유무 | `[데이터명] 있음/없음` | `추천상품 있음`<br>`이벤트배너 없음` |
| 사용자 상태 | `[상태] 사용자` | `로그인 사용자`<br>`VIP 사용자`<br>`비로그인 사용자` |
| 값 비교 | `[필드명] [조건] [값]` | `할인율 10% 이상`<br>`재고 5개 이하`<br>`가격 10000원 이상` |
| 기간 | `[시작일]~[종료일]` | `2026-05-01 ~ 2026-05-31` |
| 항상 노출 | `항상` | `항상` |

#### 복합 조건 작성
```
노출 조건:
  - 로그인 사용자
  - 추천상품 1개 이상
```

### 3.4 이벤트/액션

#### Organism-SB 테이블 작성

| no. | 컴포넌트 ID | 컴포넌트 명 | 이벤트 | 액션 | 비고 |

### 3.7 법적 고지·안내 작성 룰 ⭐ (2026-05-01 신설)

> 📍 **source-of-truth**: `mockup/mockup-docs/POLICY_REPORT_TEMPLATE.md § 1.8` (마킹 표준 + 고지 대상 프로세스 + 표시 위계) + `mockup/mockup-docs/SPEC_REPORT_TEMPLATE.md § 0.5` (법적·안내 요소 체크). 본 § 은 SDUI description 태그 변환 가이드.

**SDUI 매핑:** description 안 `[고지:]` 태그 (description-format.md 참조)

법적 의무 또는 정책서 명시 안내는 **빠지면 컴플라이언스 위반** 가능. 모든 SB 의 description 검토 시 다음 6 카테고리 키워드 검색 후 누락 없는지 확인.

#### 의무 표시 카테고리

| 카테고리 | 표시 의무 | description 태그 마킹 |
|---|---|---|
| **동의 화면** (약관/마케팅/개인정보) | 필수/선택 배지 + 전체동의 적용 범위 명시 | `[고지:필수\|PG-...] 필수 약관 동의` |
| **고위험 행위** (탈퇴/강제 제한) | "되돌릴 수 없음" / "복구 불가능" 명시 | `[고지:필수\|PR-...-CS-...] 탈퇴 영향 — 복구 불가` |
| **상태 제한** (휴면 해제 불가, 가입 제한) | 사유 + 해제일/예상일 + 문의처 | `[고지:필수\|PG-...] 제한 사유 + 해제일 + 1644-0009` |
| **본인인증 정보 수집** (휴대폰/PASS) | 수단별 수집 정보 범위 inline | `[고지:필수\|PG-...-AUTH-...] 수집 정보: 이름·생년월일·휴대폰·CI` |
| **유예 기간** (탈퇴 7일 / 인증 시간) | 기한 + 만료 후 결과 명시 | `[고지:필수\|PR-...-CS-003-06] 7일 이내 철회 가능 / 이후 복구 불가` |
| **고객센터 footer** (전 화면 sticky) | 1644-0009 + 1:1 문의 link | `[고지:필수\|전기통신사업법 § 22-2] 고객센터 노출` |

#### 마킹 표준

| 마킹 | 정의 | 누락 시 |
|---|---|---|
| `필수` | 법적 의무 (전기통신사업법 § 22-2 / 개인정보보호법) **또는** 정책서에 "안내·고지" 형태로 명시 | 컴플라이언스 위반 — 자동 reject |
| `사용성` | 정책서엔 추상적, UX 보완 | 리뷰 권장 |

#### 키워드 검색 체크리스트 (정책서 → SB 작성 시)

- [ ] **동의 / 약관 / 고지** — 동의 화면 위계 (필수/선택/전체)
- [ ] **필수 / 선택** — 동의 항목 분기 → 시각 차별화
- [ ] **안내 / 주의 / 유의 / 경고** — 배너·모달·인라인
- [ ] **불가 / 불가역 / 복구 / 되돌릴 수** — 강 톤 텍스트
- [ ] **휴면 / 탈퇴 / 제한 / 차단** — 상태별 후속 안내
- [ ] **본인인증 / 통신사 / 정보 수집 / 보유 기간** — 개인정보 고지
- [ ] **문의 / 고객센터 / 1:1 / 연락처** — 항상 노출 위치
- [ ] **유예 / 철회 / 기한 / 만료일** — 기간 제한 안내

#### 작성 예시

```
[영역명] 약관 동의 list
[고지:필수|PG-MBR-TERM-001] 필수/선택 약관 분기 — 전기통신사업법 § 22-2 의무 표시
[조건:필수약관미동의] [다음] disabled + 미동의 row 강조
[조건:전체동의체크] 적용 범위 명시 ("필수+선택 모두" 또는 "필수만 동의함")
[상태:loading] skeleton row × N + [다음] disabled
[상태:empty] "현재 동의할 약관이 없습니다" 안내 + 자동 진행
[상태:error] snack-bar "약관을 불러오지 못했습니다" + [재시도]
```

> 📍 **source-of-truth**: `mockup/mockup-docs/SPEC_REPORT_TEMPLATE.md § 1.6` (상태 매트릭스) + `SPEC_INPUT_TEMPLATE.md § 컨텍스트 > 상태 시나리오` (default/loading/empty/error/partial/branch 6 상태). 본 § 은 SDUI description 태그 변환 가이드.

**SDUI 매핑:** description 안 `[상태:]` 태그 (description-format.md 참조)

화면이 운영 시 마주칠 모든 상태 (default 외) 를 SB 작성 단계에서 명시. 누락 시 운영에서 빈 화면 / generic placeholder 노출 위험.

#### 필수 명시 상태 (6종)

| 상태 | 트리거 | description 작성 |
|---|---|---|
| `default` | 정상 진입 | 기본 영역 구성 (생략 가능) |
| `loading` | API 호출 중 | skeleton / 로딩 인디케이터 / CTA disabled |
| `empty` | 데이터 응답 0 | 안내 텍스트 + 대체 액션 |
| `error` | API 4xx/5xx | snack-bar / 재시도 버튼 / 영역 숨김 |
| `partial` | 일부 입력만 됨 | 미입력 영역 강조 + CTA 분기 |
| `branch` | 분기 케이스 (미성년·제한 등) | 별도 화면 / BottomSheet 안내 |

#### 작성 예시

```
[영역명] 검색 결과 list
[상태:default] list-item × N (정확 매치 + 관련 결과)
[상태:loading] skeleton list-row × 3 (정확 매치 영역)
[상태:empty] zero 화면으로 redirect ("검색 결과 없음" hero + 대체 키워드)
[상태:error] snack-bar "검색 실패" + [재시도] 버튼
```

## 5. 설계서 변환 원칙 (정책서 → SDUI 화면설계서)

### 5.1 Screen 도출 규칙

**핵심 원칙: `1 Process = 1 Screen`**

정책서의 프로세스(PR-*) 1개가 Screen-SB 1개에 대응한다. UseCase 단위가 아니라 Process 단위로 화면을 도출한다.

#### 화면 도출 판정 기준

| 판정 | 조건 | 예시 |
|------|------|------|
| **Screen으로 도출** | 고객이 직접 보거나 조작하는 UI가 필요한 프로세스 | 약관 동의, 본인인증, 사유 입력, 결과 안내 |
| **Screen에서 제외** | 백엔드 자동 처리만으로 완결되는 프로세스 | 회원 검증 (BSS 자동 판정), 가입 처리 (계정 생성) |
| **인접 Screen에 합침** | 제외 대상이지만 결과 UI가 필요한 경우, 직후 Screen에 로딩/결과 영역으로 포함 | 회원 검증 → 가입 완료 화면의 로딩 상태로 합침 |

#### TYPE_CODE 결정 기준

| 프로세스 성격 | TYPE_CODE | 판정 근거 |
|-------------|-----------|----------|
| 전체 화면 폼/목록/안내 | `PG` (Page) | 하단 네비 포함, 독립 화면 |
| 단계형 플로우 (가입, 인증 등) | `PG` 또는 `FM` (FullModal) | 하단 네비 없이 단계 진행이면 FM |
| 확인/취소 의사 결정 | `CP` (ConfirmPopup) | "탈퇴 최종 동의" 등 2버튼 팝업 |
| 상태 알림 (1버튼) | `AP` (AlertPopup) | "휴면 상태입니다" 등 알림 |
| 처리 중 대기 | `LD` (LoadingOverlay) | 백엔드 처리 대기 |

#### 재사용 Screen 규칙

동일한 UI 구조가 context만 다른 경우 (예: 본인인증이 가입/탈퇴/휴면해제에서 반복), **1개 Screen-SB를 작성**하고 다른 화면에서 참조한다.

- Screen-SB 메타데이터에 `재사용 대상` 필드를 추가
- 참조하는 Screen-SB에서 `연관 설계서`로 링크
- context별 차이는 `노출 조건` 또는 `서버 제어 항목`으로 분기

### 5.2 변환 단계

| 단계 | 작업 | 기준 |
|------|------|------|
| 1. IA 도출 | 정책서 프로세스(PR-*)를 화면으로 변환 | 1 Process = 1 Screen (백엔드 전용 제외) |
| 2. 화면 판정 | 각 프로세스의 Screen 도출 여부 결정 | 위 판정 기준 적용 |
| 3. TYPE_CODE 결정 | 프로세스 성격에 맞는 구현 유형 부여 | 위 TYPE_CODE 기준 적용 |
| 4. 화면 ID 부여 | NOVA 규칙으로 ID 할당 | UseCase 내 여정 순서대로 NNN 부여 |
| 5. 섹션 경계 | 동일 API/데이터 소스를 공유하는 Organism 묶음 | 재사용 섹션은 context param으로 분기 |
| 6. Organism 정의 | 노출 조건, 데이터 바인딩, 이벤트, 텍스트 규칙 | Molecule/Atom은 디자인 시스템 참조만 |
| 7. 갭 식별 | 정책서에 없는 플로우/값/조건 마킹 | G-NN 번호로 추적 |
