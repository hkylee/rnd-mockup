2.2 스키마 버전 및 Renderer 호환성


현재 스키마 버전: 1.0.0

스키마 버전은 Semantic Versioning을 따릅니다:

  • Major: 하위 호환성이 없는 변경
  • Minor: 하위 호환성을 유지하는 기능 추가
  • Patch: 하위 호환성을 유지하는 버그 수정

2.2.1 Renderer 호환성 선언


SDUI 스키마는 자신을 렌더링하기 위해 필요한 최소 sdui-renderer 버전을 minRendererVersion 필드로 명시합니다.

{

  "version": "1.5.0",

  "minRendererVersion": "1.2.0",

  "metadata": { ... },

  "children": [ ... ]

}


version

string

✅

-

스키마 자체 버전 (문서화 및 추적용)

minRendererVersion

string

-

"1.0.0"

이 스키마를 렌더링하기 위한 최소 renderer 버전

호환성 검사 원리:

  • sdui-renderer는 자신의 버전(VERSION_CONFIG.renderer)을 알고 있음
  • 스키마 로드 시 minRendererVersion과 renderer 버전을 비교
  • renderer 버전이 minRendererVersion보다 낮으면 에러 발생

Renderer Version  vs  minRendererVersion

     1.3.0       >=       1.2.0           → ✅ 호환 (렌더링 진행)

     1.1.0       <        1.2.0           → ❌ 비호환 (에러 + 업그레이드 권고)

설계 근거:

스키마는 새로운 기능이나 문법을 사용할 때마다 minRendererVersion을 올립니다. 이미 배포된 renderer는 미래의 스키마 버전을 예측할 수 없으므로, 스키마가 "나를 렌더링하려면 최소 X 버전의 renderer가 필요해"라고 선언하는 방식이 합리적입니다.

2.3 디자인 토큰 아키텍처


SDUI Engine은 디자인 토큰 기반 스타일링을 사용합니다. 픽셀 값 대신 토큰 이름을 사용하여 일관된 디자인 시스템을 유지합니다.

2.3.1 토큰 번들링 구조


SDUI Engine Core (npm 패키지)

├── @skt/components        # UI 컴포넌트 라이브러리

└── @skt/design-tokens     # 디자인 토큰 (spacing, colors, typography)

  • 디자인 토큰은 Component Library와 함께 번들링되어 배포됩니다
  • SDUI Engine은 토큰 이름을 실제 CSS 값으로 해석합니다

5.2 className 필드


컴포넌트에 추가적인 Tailwind CSS 클래스를 적용할 때 사용합니다.

{

  "type": "Card",

  "id": "card-1",

  "className": "shadow-lg hover:shadow-xl transition-shadow",

  "props": {

    "title": "상품 카드"

사용 지침:

  • 디자인 토큰으로 표현하기 어려운 세밀한 스타일에만 사용
  • 컴포넌트 기본 스타일과 충돌하지 않도록 주의

5.3 style 필드 (공통 스타일 속성)


컴포넌트에 inline CSS로 적용되는 공통 스타일 속성을 정의합니다. style 필드의 속성들은 SDUI Renderer가 wrapper div의 style에 적용합니다.

v1.13.0 변경: style은 backgroundColor와 opacity만 포함합니다. 기타 스타일 속성(margin, padding, width, borderRadius, shadow 등)은 className(Tailwind CSS)으로 관리합니다.
v1.17.0 변경: backgroundColor → background로 키 변경. CSS background shorthand를 통해 색상, 그라디언트, 이미지를 단일 문자열로 표현합니다.

5.4 props 필드와 데이터 바인딩


props의 각 속성은 정적 값, 템플릿 문자열 ({{...}}), 또는 바인딩 객체 ({ bind, default })로 지정할 수 있습니다.

type PropValue<T> =

  | T // 정적 값

  | string // 템플릿 문자열 ({{...}} 포함 시)

  | { bind: string; default?: T }; // 바인딩 객체 (폴백 지원)

{

  "type": "Button",

  "props": {

    "variant": "primary", // 정적 값

    "text": "{{button.label}}", // 템플릿 문자열

    "disabled": {

      // 바인딩 객체 (폴백 포함)

      "bind": "{{cart.isEmpty}}",

      "default": false

    }

  }

}

상세 설명: § 6. 데이터 바인딩 참조

5.9 컴포넌트 중첩 규칙


일반적으로 Layout 컴포넌트(Container, Section 등)는 다른 Layout 컴포넌트를 children으로 포함할 수 없습니다. 단, Section 컴포넌트는 예외적으로 children에 다른 Section을 포함할 수 있습니다 (재귀적 중첩 허용).


6. 데이터 바인딩 (Binding)


6.1 바인딩 문법


데이터 바인딩은 {{expression}} 형식의 Mustache 스타일 문법을 사용합니다. 바인딩은 props 내에서 직접 정의합니다.

6.2 바인딩 표현 방식


props의 각 속성은 정적 값, 템플릿 문자열, 또는 바인딩 객체로 지정할 수 있습니다.

type PropValue<T> =

  | T // 정적 값

  | string // 템플릿 문자열 ({{...}} 포함 시)

  | {

      bind: string; // 바인딩 표현식

      default?: T; // 폴백 값 (선택)

    };


정적 값

"text": "구매하기"

-

고정 값 표시

템플릿 문자열

"text": "{{product.name}}"

❌

최대한 바인딩 (없으면 빈 문자열)

바인딩 객체

{ "bind": "{{product.name}}", "default": "상품" }

✅

모든 필드 유효해야 바인딩, 아니면 default

단순 바인딩


{

  "type": "Button",

  "props": {

    "text": "{{button.label}}" // 템플릿 문자열

  }

}

폴백이 필요한 경우


{

  "type": "ProductCard",

  "props": {

    "title": {

      "bind": "{{product.name}}",

      "default": "상품명"

    }

  }

}

복합 템플릿 (여러 {{...}} 조합)

// default 있음: 모든 필드 유효해야 바인딩

{

  "text": {

    "bind": "{{user.name}}님, 포인트: {{user.point}}P",

    "default": "로그인 필요"

  }

}

// user = { name: "홍길동", point: 0 } → "홍길동님, 포인트: 0P" (0은 유효)

// user = { name: "홍길동" } → "로그인 필요" (point 없음)


// default 없음: 최대한 바인딩

{

  "text": "{{user.name}}님, 포인트: {{user.point}}P"

}

// user = { name: "홍길동" } → "홍길동님, 포인트: P"

유효성 판단: undefined, null만 무효. 0, "", false는 유효한 값.

6.5 예약어 (Reserved Keywords)


event 접두어는 런타임 이벤트 컨텍스트 전용 예약어로, 데이터 키로 사용할 수 없습니다.

6.5.1 event.* 예약어


event.*는 이벤트 핸들러 내에서 JavaScript 표준 DOM Event 객체의 모든 속성에 접근할 수 있는 특수 컨텍스트입니다. 데이터 컨텍스트(스키마 data 또는 SDUIRenderer data props)에서 찾을 수 없으며, 이벤트가 발생할 때 런타임에 주입됩니다.

주요 사용 예시:

{{event.target.value}}

입력 필드의 현재 값

onChange 이벤트 핸들러

{{event.target.checked}}

체크박스의 체크 상태

onChange 이벤트 핸들러

{{event.key}}

눌린 키보드 키 (예: Enter)

onKeyDown 이벤트 핸들러

{{event.clientX}}

마우스 X 좌표 (뷰포트)

onClick 이벤트 핸들러

{{event.clientY}}

마우스 Y 좌표 (뷰포트)

onClick 이벤트 핸들러

{{event.type}}

이벤트 타입 (예: click)

모든 이벤트 핸들러

위 목록은 자주 사용되는 예시이며, JavaScript DOM Event 객체가 제공하는 모든 속성을 동일한 방식으로 사용할 수 있습니다.

사용 예시:

{

  "events": {

    "onChange": [

      {

        "action": "setState",

        "key": "searchKeyword",

        "value": "{{event.target.value}}"

      }

    ]

  }

}

6.5.2 예약어 제한 사항


다음 키는 데이터 컨텍스트에서 사용 금지됩니다:

// ❌ 금지: event 접두어를 데이터 키로 사용

{

  "data": {

    "event": { ... },

    "eventData": { ... }

  }

}


// ❌ 금지: setState의 key로 event 접두어 사용

{

  "action": "setState",

  "key": "event",

  "value": "..."

}


// ✅ 허용: 다른 이름 사용

{

  "data": {

    "eventInfo": { ... },

    "myEvent": { ... }

  }

}

Renderer 동작: event로 시작하는 키가 데이터에 정의되면 무시됩니다.

7.2 커스텀 이벤트 (컴포짓 컴포넌트용)


컴포짓 컴포넌트(예: ProductCard)는 내부 요소별로 커스텀 이벤트를 노출할 수 있습니다. 커스텀 이벤트는 표준 이벤트와 동일한 방식으로 events 필드에 정의합니다.

onBuyClick

구매 버튼 클릭

onWishClick

찜하기 버튼 클릭

onShareClick

공유 버튼 클릭

onTabChange

탭 변경 시

onItemSelect

아이템 선택 시
네이밍 규칙: 커스텀 이벤트는 on + 동작/요소명 + 이벤트 형식으로 명명합니다. 예: onBuyClick, onWishToggle, onItemSelect

7.3 이벤트 스키마


{

  "events": {

    "[eventType]": [SDUIAction, SDUIAction, ...]

  }

}

  • eventType은 표준 이벤트(onClick, onChange 등) 또는 커스텀 이벤트(onBuyClick 등)
  • 이벤트 핸들러는 항상 배열 형식으로 정의합니다. 단일 액션도 배열로 감싸야 합니다.

7.4 예시: 표준 이벤트


{

  "events": {

    "onClick": [

      { "action": "navigate", "url": "/purchase" },

      { "action": "track", "event": "purchase_clicked" }

    ],

    "onChange": [

      {

        "action": "setState",

        "key": "selectedValue",

        "value": "{{event.target.value}}"

      }

    ]

  }

}

7.5 예시: 컴포짓 컴포넌트 커스텀 이벤트


{

  "type": "ProductCard",

  "id": "product-1",

  "props": {

    "product": { "bind": "{{product}}" }

  },

  "events": {

    "onClick": [{ "action": "navigate", "url": "/product/{{product.id}}" }],

    "onBuyClick": [

      {

        "action": "apiCall",

        "method": "POST",

        "endpoint": "/api/cart/add",

        "body": { "productId": "{{product.id}}" }

      }

    ],

    "onWishClick": [

      {

        "action": "apiCall",

        "method": "POST",

        "endpoint": "/api/wish/toggle",

        "body": { "productId": "{{product.id}}" }

      }

    ],

    "onShareClick": [

      {

        "action": "custom",

        "handler": "showShareDialog",

        "params": { "productId": "{{product.id}}" }

      }

    ]

  }

}

컴포짓 컴포넌트 이벤트 처리 방식:
SDUI Renderer는 컴포넌트 내부 구조를 알 수 없으므로(블랙박스), 컴포짓 컴포넌트가 이벤트 핸들러를 props로 노출하면 Renderer가 events에서 해당 이벤트를 찾아 연결합니다.
  • 컴포넌트는 onXxxClick 형태의 이벤트 핸들러 props를 정의
  • SDUI JSON의 events 필드에 해당 이벤트명으로 액션 배열 정의
  • Renderer가 events를 스캔하여 컴포넌트 props로 전달

9.5 조건부 렌더링 (Display)


9.5.1 개요


display 필드는 데이터 값에 따라 컴포넌트의 표시/숨김을 동적으로 제어합니다.

주요 사용 사례:

  • 로그인 상태에 따른 UI 분기
  • 사용자 권한(프리미엄, 일반 등)별 컴포넌트 표시
  • 동적 폼 필드 표시/숨김
  • 사용자 인터랙션에 따른 UI 변경

9.5.2 스키마 정의


interface SDUIDisplay {

  /** 조건부 렌더링 조건 */

  when?: PropBinding | boolean;

}


when

PropBinding | boolean

-

-

조건식 (truthy면 표시, falsy면 숨김)

9.5.3 동작 원리


  • display.when이 truthy → 컴포넌트 렌더링
  • display.when이 falsy → null 반환 (DOM에서 제거)
  • display 필드 없음 → 항상 표시 (기존 동작 유지)

9.5.4 기본 사용 예시


예시 1: 로그인 상태에 따른 버튼 표시


{

  "type": "Button",

  "id": "mypage-btn",

  "display": {

    "when": { "bind": "user.isLoggedIn", "default": false }

  },

  "props": {

    "variant": "primary",

    "children": "마이페이지"

  },

  "events": {

    "onClick": [{ "action": "navigate", "url": "/mypage" }]

  }

}

예시 2: 프리미엄 사용자 전용 컨텐츠


{

  "type": "Container",

  "id": "premium-section",

  "display": {

    "when": { "bind": "user.isPremium" }

  },

  "children": [

    {

      "type": "Text",

      "id": "premium-text",

      "props": {

        "children": "프리미엄 전용 혜택"

      }

    }

  ]

}

9.5.5 동적 상태와 연동


setState 액션과 함께 사용하여 사용자 인터랙션에 따라 UI를 동적으로 변경할 수 있습니다.

{

  "version": "1.9.0",

  "metadata": {

    "title": "상세보기 토글 예시"

  },

    "showDetails": false

  },

  "children": [

      "type": "Button",

      "id": "toggle-btn",

      "props": {

        }

      },

      "events": {

        "onClick": [

          {

            "action": "setState",

            "key": "showDetails",

            "value": { "bind": "showDetails", "transform": "!showDetails" }

          }

        ]

      }

    },

    {

      "type": "Container",

      "id": "details-panel",

      "display": {

        "when": { "bind": "showDetails" }

      },

      "className": "p-4 bg-gray-50 rounded-md",

      "children": [

        {

          "type": "Text",

          "id": "details-text",

          "props": {

            "children": "여기에 상세 정보가 표시됩니다."

    }

9.5.7 default 값 활용


바인딩된 데이터가 없을 때의 기본 동작을 정의할 수 있습니다.

{

  "type": "Button",

  "id": "feature-btn",

  "display": {

    "when": {

      "bind": "features.newFeatureEnabled",

      "default": false

    }

  "props": {

    "children": "신규 기능"

  }

}

9.5.8 주의사항


  1. 성능 고려: display.when이 자주 변경되는 값을 참조하면 리렌더링이 빈번하게 발생할 수 있습니다.
  2. 접근성: 조건부로 숨겨진 컴포넌트는 DOM에서 완전히 제거되므로, 스크린 리더에서도 접근할 수 없습니다.
  3. 애니메이션: 현재 버전(1.9.0)은 진입/퇴장 애니메이션을 지원하지 않습니다. 즉시 표시/숨김 처리됩니다.

11.6 PropValue


/**

 * props의 각 속성 값 타입

 * - 정적 값: T 타입 그대로

 * - 바인딩 객체: { bind, default? } 형태

 */

type PropValue<T = unknown> =

  | T

  | {

      bind: string; // 바인딩 표현식 ({{...}} 형식)

      default?: T; // 바인딩 실패 시 폴백 값

    };


/**

 * 바인딩 객체 타입 가드

 */

function isBindingValue(

  value: unknown

): value is { bind: string; default?: unknown } {

  return typeof value === 'object' && value !== null && 'bind' in value;

}

12.1 CustomArea 컴포넌트 스펙


12.1.1 개요


CustomArea는 SDUI 스키마 내에서 외부 React 컴포넌트를 주입할 수 있는 특수 컴포넌트입니다. 서버가 정의한 레이아웃 구조 내에 클라이언트 앱의 네이티브 컴포넌트를 삽입할 수 있습니다.

12.1.2 사용 사례


  • 헤더/푸터 영역: 앱별 커스텀 헤더/푸터 컴포넌트 주입
  • 사용자 프로필: 로그인 상태에 따른 동적 UI
  • 결제 모듈: 앱별 결제 UI 통합
  • 광고 배너: 클라이언트 관리 광고 영역
  • 플랫폼 의존 기능: OS별 네이티브 기능 UI

12.1.3 스키마 정의


{

  "type": "CustomArea",

  "id": "header-area"

}

12.1.4 클라이언트 사용 방법


import { SDUIRenderer } from '@skt-mno/sdui-renderer';

import { CustomHeader } from './components/CustomHeader';

import { CustomFooter } from './components/CustomFooter';


function App() {

  return (

    <SDUIRenderer

      id="main-page"

      customAreas={{

        'header-slot': <CustomHeader />,

        'footer-slot': <CustomFooter />,

      }}

    />

  );

}
